# 企业文档中英互译系统技术穿刺清单

## 1. 背景与目标

本清单基于 `docs/DESIGN.md` 的总体设计整理，目标不是重复架构描述，而是把“企业文档中英互译 / 英中互译 + PDF/DOCX 保真 + 企业背景注入 + 内网大模型”的关键风险，转化为可执行的技术穿刺与优化项。

本清单主要回答三个问题：

1. 一期要先验证哪些关键能力，才能证明路线可行
2. 哪些问题属于必须先穿刺的交付风险，不能等到后期再补
3. 哪些能力适合放在一期，哪些应明确放到后续阶段

---

## 2. 冲刺原则

### 2.1 先保结构，再提翻译质量

对于企业级文档翻译，用户首先感知的是“能不能交付”，而不是模型是否给出了漂亮译文。因此优先级应始终是：

1. 结构不坏
2. 版式不炸
3. 术语不漂
4. 风格统一
5. 译文进一步优化

### 2.2 先 DOCX，再数字 PDF，再复杂 PDF

建议一期优先证明：

- DOCX 高保真翻译主链路可交付
- 数字 PDF 可稳定解析并导出 DOCX

不建议一期承诺：

- 扫描 PDF 高保真翻译
- 翻译后直接输出强保真的目标 PDF
- 坐标级 PDF 回填

### 2.3 先做可交付，再追求极致自动化

优先建设：

- 自动质量校验
- 失败页重跑
- 高风险节点标记
- 任务可恢复

而不是一开始追求：

- 全流程多智能体自主决策
- 复杂文档零人工干预
- 所有格式一次性全支持

### 2.4 多智能体优先用于分析、质检、学习

多智能体更适合放在：

- 全文分析
- 企业背景与对象确认
- 行业术语整理
- 质量复核
- 术语学习与冲突审计

不建议一期把多智能体作为主翻译执行架构。

---

## 3. 优先级定义

- **P0**：必须优先验证，否则无法形成可交付主链路
- **P1**：紧随 P0 的关键增强项，直接影响专业度和规模化稳定性
- **P2**：后续扩展项，影响支持边界和产品上限，但不阻断一期 MVP

---

## 4. P0 冲刺清单

### P0-1 DOCX 高保真翻译主链路

- [ ] **目标**：验证 DOCX -> 结构化解析 -> 翻译 -> DOCX 回填 是否可交付
- [ ] **为什么重要**：DOCX 是当前最可控的企业文档格式，也是最适合优先验证交付能力的主链路
- [ ] **关键风险**：
  - 段落样式回填不稳定
  - 标题层级丢失
  - 表格结构损坏
  - 页眉页脚、首页设置丢失
  - 混合 run 样式被破坏
- [ ] **验证方法**：
  - 选 3~5 份典型 DOCX 样本，覆盖正文、标题、表格、页眉页脚、封面
  - 比对源文档与目标文档的结构一致性、样式一致性、内容一致性
- [ ] **建议样本**：
  - 企业白皮书
  - 用户手册
  - 合同/制度类文档
  - 带复杂表格的商务文档
- [ ] **通过标准**：
  - 标题、表格、页眉页脚、首页保留稳定
  - 主要段落样式不明显退化
  - 输出文档可继续编辑，且不需要大范围返工
- [ ] **可能产出的工程项**：
  - run 级 IR 增强
  - 段落/单元格回填优化
  - 首页与页眉页脚专项回归集

### P0-2 数字 PDF -> DOCX 可交付性

- [ ] **目标**：验证数字 PDF 是否可以作为一期主链路输入，并稳定输出可编辑 DOCX
- [ ] **为什么重要**：用户实际上传的大量企业资料会是 PDF，若数字 PDF 主链路不成立，产品边界必须重定
- [ ] **关键风险**：
  - 阅读顺序错误
  - 多栏混排失真
  - 表格识别失败
  - 图片/图表位置错乱
  - 页眉页脚无法稳定识别
- [ ] **验证方法**：
  - 选 3 类 PDF：规则单栏、复杂多栏、年报类
  - 评估解析质量、导出 DOCX 可读性、可编辑性、人工修复成本
- [ ] **建议样本**：
  - 年报 PDF
  - 产品白皮书 PDF
  - 合规/制度 PDF
- [ ] **通过标准**：
  - 规则数字 PDF 基本可交付
  - 复杂 PDF 至少能明确降级边界
  - 输出 DOCX 可被人工低成本修订
- [ ] **可能产出的工程项**：
  - PDF 分类器
  - 阅读顺序恢复增强
  - 页眉页脚区域识别
  - 页面块到 DOCX 映射优化

### P0-3 翻译后排版与长度膨胀风险

- [ ] **目标**：验证中英互译导致的长度变化是否会破坏标题、表格、封面和页眉页脚布局
- [ ] **为什么重要**：排版问题往往比语言问题更早暴露为“不可交付”
- [ ] **关键风险**：
  - 英文标题变长后换行异常
  - 表格单元格溢出
  - 封面主标题/副标题错位
  - 页眉页脚挤压
  - 列表编号错位
- [ ] **验证方法**：
  - 统计源文与译文长度比
  - 对标题、封面、表格单元格建立高风险样本集
  - 做回填后的视觉检查和规则检查
- [ ] **建议样本**：
  - 封面页
  - 财务表格页
  - 多级标题页
  - 短页眉页脚页
- [ ] **通过标准**：
  - 能识别高风险节点
  - 高风险节点可触发告警或短译策略
  - 大多数页面无明显排版炸裂
- [ ] **可能产出的工程项**：
  - 排版风险评分器
  - 标题/表格短译策略
  - 回填后溢出检测

### P0-4 企业背景注入 A/B 效果

- [ ] **目标**：验证企业信息、企业背景、企业习惯注入后，译文质量是否显著提升
- [ ] **为什么重要**：这是该产品区别于通用翻译器的核心价值点
- [ ] **关键风险**：
  - prompt 变长但质量无明显提升
  - 背景信息引入事实污染
  - 不同企业的风格模板泛化能力差
- [ ] **验证方法**：
  - 对同一批段落做 A/B 测试：
    - A：仅段落与邻近上下文
    - B：段落 + Document Analysis Brief + 企业档案 + Object Dossier + Style Guide
  - 比较术语稳定性、对象识别正确率、文风一致性、人工可接受度
- [ ] **建议样本**：
  - 企业年报
  - 产品介绍
  - 公司制度文件
- [ ] **通过标准**：
  - 公司名、产品名、对象代称更稳定
  - 企业文风更统一
  - 人工 review 明显感知提升
- [ ] **可能产出的工程项**：
  - Enterprise Profile 数据结构
  - 项目级企业档案配置
  - 文档级对象信息卡生成

### P0-5 术语一致性穿刺

- [ ] **目标**：验证术语表、对象信息卡、style guide 是否足以压住整本文档术语漂移
- [ ] **为什么重要**：企业文档交付里最常见的问题是同一术语前后译法不一致
- [ ] **关键风险**：
  - 同一术语跨页多译
  - 标题与正文译法不一致
  - 表格与正文译法不一致
  - 页眉页脚与正文不一致
- [ ] **验证方法**：
  - 建立术语追踪样本
  - 翻译后做多译法聚类分析
  - 对比是否命中 glossary / do_not_translate / object dossier
- [ ] **建议样本**：
  - 术语重复出现频率高的技术文档
  - 年报、保险、合规类文档
- [ ] **通过标准**：
  - 高频术语基本无漂移
  - 公司名、产品名、模块名策略稳定
  - 质量校验能自动发现高风险冲突
- [ ] **可能产出的工程项**：
  - 术语一致性检查器
  - segment 级术语命中日志
  - 高风险术语自动阻断规则

### P0-6 封面 / 首页专项

- [ ] **目标**：验证封面、首页、标题、副标题、公司名、版本号等关键内容的翻译与回填质量
- [ ] **为什么重要**：首页是用户最敏感的区域，哪怕正文质量不错，首页出错也会直接拉低整份文档观感
- [ ] **关键风险**：
  - 公司名策略错误
  - 标题过长导致居中布局破坏
  - 版本号、年份、法律说明误译
- [ ] **验证方法**：
  - 首页单独走 `cover_profile`
  - 单独评估语言质量、术语策略、版式稳定性
- [ ] **建议样本**：
  - 年报封面
  - 白皮书封面
  - 合规文档首页
- [ ] **通过标准**：
  - 首页关键字段无明显错误
  - 标题观感稳定
  - 可独立输出首页风险报告
- [ ] **可能产出的工程项**：
  - 首页优先处理链路
  - 首页阻断式质检
  - 首页预览产物

### P0-7 表格专项翻译与回填

- [ ] **目标**：验证表格是否可以作为可交付的一期对象，而不是纯文本降级处理
- [ ] **为什么重要**：企业文档中表格通常承载最关键的信息，且最容易因为翻译长度变化导致布局崩坏
- [ ] **关键风险**：
  - 合并单元格丢失
  - 跨页表头错乱
  - 数值和单位丢失
  - 单元格过长导致表格不可读
- [ ] **验证方法**：
  - 建表格专项样本集
  - 关注表头、行头、金额、百分比、单位、跨页情况
- [ ] **建议样本**：
  - 财务报表
  - 参数对照表
  - 合规披露表
- [ ] **通过标准**：
  - 表结构完整
  - 数字和单位无偏差
  - 表头/行头语义清晰
  - 溢出风险可自动告警
- [ ] **可能产出的工程项**：
  - `table_profile`
  - 表格长度/溢出检查器
  - 跨页表头拼接与回填增强

### P0-8 数字 / 单位 / 日期 / 编号保真

- [ ] **目标**：验证翻译过程中数字与关键格式信息是否严格保真
- [ ] **为什么重要**：企业文档里，事实错误比措辞错误更不可接受
- [ ] **关键风险**：
  - 金额、百分比、小数位变动
  - 日期格式误改
  - 法规编号、章节编号、版本号丢失
  - 列表序号错乱
- [ ] **验证方法**：
  - 预处理阶段抽取关键 token
  - 翻译后做逐项比对
  - 不一致时自动阻断或重跑
- [ ] **建议样本**：
  - 财务表格
  - 监管/合规条文
  - 操作手册编号页
- [ ] **通过标准**：
  - 数字与单位错误率接近 0
  - 编号与版本号严格保留
  - 关键差异能被自动发现
- [ ] **可能产出的工程项**：
  - 数字/单位保护器
  - 关键 token 锁定机制
  - 事实保真阻断规则

## 5. P1 冲刺清单

### P1-1 公司名 / 品牌名 / 产品名翻译策略

- [ ] **目标**：验证公司法定名称、官方中英文名称、品牌名、产品名、模块名的翻译策略是否可稳定执行
- [ ] **关键风险**：
  - 公司名误翻或过度本地化
  - 法定名称、证券简称、品牌名混用
  - 首次出现策略与后续简称策略不统一
- [ ] **验证方法**：
  - 建对象样本集
  - 验证官方名称确认、别名归并、保留不译、首次双写等策略
- [ ] **通过标准**：
  - 对象翻译策略稳定可追溯
  - evidence links 足以支撑译法决策
- [ ] **可能产出的工程项**：
  - Object Dossier 增强
  - 公司名策略决策器
  - 对象证据缓存

### P1-2 全文分析与 Prompt Bundle 稳定性

- [ ] **目标**：验证 `document_analysis_brief`、`object_dossier`、`industry_glossary`、`style_guide` 是否稳定、可复用、可提高翻译效果
- [ ] **关键风险**：
  - 总结结果不稳定
  - 后半文档信息丢失
  - 行业判断和目标读者判断偏差
  - prompt bundle 过大却贡献有限
- [ ] **验证方法**：
  - 同一文档重复运行多次
  - 对比产出稳定性和后续翻译质量提升幅度
- [ ] **通过标准**：
  - 关键字段波动可控
  - 输出结构化结果可直接复用
  - 能支撑后续段级翻译和质检
- [ ] **可能产出的工程项**：
  - `translation_prompt_react_agent.py`
  - prompt bundle store
  - chunk summary 评估集

### P1-3 长文档分块总结一致性

- [ ] **目标**：验证长文档采用 chunk summary -> section summary -> global summary 的层级总结后，是否会丢失关键上下文
- [ ] **关键风险**：
  - 后半段主题丢失
  - 文档多主题时误判行业
  - 不同 chunk 之间对象别名未合并
- [ ] **验证方法**：
  - 用 100 页以上文档做层级总结实验
  - 对比人工摘要与自动摘要覆盖度
- [ ] **通过标准**：
  - 章节主线覆盖较完整
  - 行业、对象、风格结论不明显漂移
- [ ] **可能产出的工程项**：
  - chunk 切分策略
  - summary 覆盖度检查器
  - 摘要一致性审查规则

### P1-4 质量校验器增强

- [ ] **目标**：从“基础校验”升级到“交付阻断级校验”
- [ ] **关键风险**：
  - 只做轻量 warning，未覆盖关键风险
  - 术语、结构、事实问题无法阻断
- [ ] **验证方法**：
  - 为术语、数字、结构、首页、表格建立专项误差样本
  - 验证 warning / blocking 的判定是否合理
- [ ] **通过标准**：
  - 关键问题可自动阻断
  - 问题页和问题段可定位、可回放
- [ ] **可能产出的工程项**：
  - 首页专项质检器
  - 术语冲突阻断
  - 溢出风险检测

### P1-5 文档可支持性分类

- [ ] **目标**：上传后先判断文档属于哪类支持边界，避免超范围文档直接进入主链路
- [ ] **关键风险**：
  - 用户把扫描 PDF 或复杂图文化 PDF 当作一期必支持输入
  - 系统输出很差但缺少前置告知
- [ ] **验证方法**：
  - 建立文档分类规则：DOCX / 数字 PDF / 扫描 PDF / 混合 PDF / 高风险版式 PDF
- [ ] **通过标准**：
  - 能较准确判断输入类型和支持等级
  - 能在创建任务时给出边界说明
- [ ] **可能产出的工程项**：
  - 文档分类器
  - 支持等级标签
  - 任务前置告警

## 6. P2 冲刺清单

### P2-1 扫描 PDF / OCR 能力

- [ ] **目标**：评估扫描 PDF 是否值得纳入后续主链路，以及 OCR + 布局重建的复杂度
- [ ] **关键风险**：
  - OCR 噪声破坏术语和数字准确性
  - 版式重建成本远高于预期
- [ ] **验证方法**：
  - 建扫描 PDF 小样本集
  - 对比 OCR 前后文本质量、布局恢复质量、人工修复成本
- [ ] **通过标准**：
  - 明确支持边界与后续投资价值
- [ ] **可能产出的工程项**：
  - OCR 接入方案
  - OCR 文本清洗
  - 扫描件专用流程

### P2-2 目录 / 书签 / 交叉引用 / 超链接

- [ ] **目标**：明确这些复杂对象在导出阶段的保留、翻译与刷新策略
- [ ] **关键风险**：
  - 目录文本与页码不一致
  - 交叉引用断裂
  - 超链接文本被误改
- [ ] **验证方法**：
  - 选包含目录、书签、超链接的样本做专项测试
- [ ] **通过标准**：
  - 明确一期降级策略或二期支持方式
- [ ] **可能产出的工程项**：
  - 目录保留/刷新策略
  - 超链接文本保护
  - 交叉引用处理规则

### P2-3 文本框 / 形状 / SmartArt / 批注 / 脚注 / 尾注

- [ ] **目标**：明确这些复杂对象的支持边界与降级策略
- [ ] **关键风险**：
  - 企业文档中这些对象出现频繁，若完全不建模，后续返工成本会很高
- [ ] **验证方法**：
  - 收集带复杂对象的 DOCX/PDF 样本
  - 判断哪些可纳入 IR，哪些应明确暂不支持
- [ ] **通过标准**：
  - 每类对象都有明确支持/降级结论
- [ ] **可能产出的工程项**：
  - `textbox` / `image_anchor` / `inline_run` 扩展
  - 复杂对象降级规则

### P2-4 翻译后目标 PDF 输出

- [ ] **目标**：评估是否需要在后续阶段支持“翻译后输出高保真目标 PDF”
- [ ] **关键风险**：
  - 坐标级回填复杂度极高
  - 目标 PDF 质量不稳定会放大交付风险
- [ ] **验证方法**：
  - 先比较“一期 PDF -> DOCX”是否已满足大多数用户需求
  - 再评估坐标级 PDF 回填的投入产出比
- [ ] **通过标准**：
  - 明确是否立项，以及何时立项
- [ ] **可能产出的工程项**：
  - 坐标级 PDF renderer
  - 页面对象定位与字体映射

## 7. 推荐执行顺序

### 阶段 A：先证明主链路可交付

建议先完成：

1. P0-1 DOCX 高保真翻译主链路
2. P0-2 数字 PDF -> DOCX 可交付性
3. P0-3 排版与长度膨胀风险
4. P0-6 封面 / 首页专项
5. P0-7 表格专项
6. P0-8 数字 / 单位 / 日期 / 编号保真

阶段 A 的目标不是把所有问题一次性解决，而是先回答：

- 一期主链路是否真实可交付
- 支持边界应如何定义
- 哪些问题必须进入阻断式质检

### 阶段 B：证明质量路线成立

建议随后完成：

1. P0-4 企业背景注入 A/B 效果
2. P0-5 术语一致性穿刺
3. P1-1 公司名 / 品牌名 / 产品名翻译策略
4. P1-2 全文分析与 Prompt Bundle 稳定性
5. P1-3 长文档分块总结一致性
6. P1-4 质量校验器增强

阶段 B 的目标是证明：

- 企业背景注入是否值得投入
- Document Analysis Brief / Object Dossier / Style Guide 路线是否成立
- 系统是否能从“能翻”走向“翻得稳定”

### 阶段 C：扩展支持边界与产品上限

建议最后评估：

1. P1-5 文档可支持性分类
2. P2-1 扫描 PDF / OCR
3. P2-2 目录 / 交叉引用 / 超链接
4. P2-3 文本框 / 脚注 / 复杂对象
5. P2-4 翻译后目标 PDF 输出

阶段 C 的目标是：

- 扩展支持边界
- 降低错误承诺
- 为二期能力建设提供清晰路线图

## 8. 一期 MVP 建议边界

建议一期 MVP 明确采用以下边界：

### 支持

- DOCX 输入
- 数字 PDF 输入
- 中文 -> 英文 / 英文 -> 中文
- 输出 DOCX
- 企业背景注入
- 对象信息卡与行业术语约束
- 首页、页眉页脚、表格、数字保真的专项校验

### 暂不承诺

- 扫描 PDF 高保真翻译
- 翻译后直接输出高保真 PDF
- 坐标级 PDF 回填
- 所有复杂对象一次性完整支持
- 多智能体主导整本文档翻译

## 10. 最终结论

该系统的一期成败，不取决于“模型能不能翻译”，而取决于下面 6 个问题能否成立：

1. DOCX 主链路是否可交付
2. 数字 PDF -> DOCX 是否可控
3. 排版风险是否可检测、可治理
4. 企业背景注入是否真实提升质量
5. 术语与对象策略是否可稳定执行
6. 首页、表格、数字等高风险区域是否可被专项治理

建议后续排期都围绕这 6 个问题推进，而不是先扩展过多格式、对象或复杂自动化能力。
