# Spike 11 最新实验根因分析

## 1. 结论先行

这次问题的核心，不是简单的“翻得不够好”，而是三类问题叠加：

1. **工程化问题是主因**  
   当前系统对内容分类、数字保护、渲染适配、特殊块处理都还不够硬，导致错误和页面质量问题持续出现。

2. **提示词问题是次因**  
   现在很多约束仍然停留在 prompt 里，导致模型既要翻译、又要理解场景、又要遵守术语、又要适应布局，负担过重，控制不稳定。

3. **大模型能力不是主因，但有边界**  
   模型不是完全不行。它已经能产出可用译文，也能遵守部分术语规则。但只靠模型和 prompt，不能稳定达到人工专业金融译者水平，尤其是在财务事实保真与正式年报文风上。

一句话判断：

> **主问题在工程化，次问题在 prompt 设计，模型能力边界是客观约束，但不是当前最优先矛盾。**

## 3. 根因一：财务事实错误，主因是工程化缺失，不是模型“笨”

### 现象
人工反馈明确指出：

- 全年总股息与末期股息混淆
- `19.97 億美元` 被错误表达为 `US$19.97 billion`
- 金额量级、数字归属、因果关系出现错误

### 根因判断
这类问题的**主因是工程化问题**，不是大模型本身不会翻译。

原因很清楚：

1. 当前系统虽然写了 prompt 规则：
   - “Do not change numbers, percentages, years, named entities or financial metric values.”
   见 `spikes/11_sanitized_background_translation/translate_with_sanitized_background.py:330-349`
2. 但系统并没有在翻译前后建立真正的数字保护机制。
3. 所以数字是否正确，本质上仍然在“赌模型是否乖”。

### 为什么这是工程化问题
因为这类内容不应该交给模型自由生成：

- 金额
- 年份
- 百分比
- 股息口径
- 指标值
- 单位换算

这些都属于**可结构化抽取、可规则校验、可自动阻断**的内容。

### 结论
> **财务事实错误的根因是：关键事实没有被工程层锁住，而是仍交给模型在自然语言生成阶段自行处理。**

### 怎么解决
#### P0 工程方案
1. 翻译前抽取并锁定：
   - 金额
   - 百分比
   - 日期
   - 股息口径
   - 财务指标
2. 翻译后逐项比对：
   - 数字值是否一致
   - 单位是否一致
   - 量级是否一致（million / billion）
   - 口径是否一致（final dividend / total dividend）
3. 不一致直接：
   - 重试
   - 或阻断该单元进入最终输出

#### 这不是优先靠 prompt 解决的问题
prompt 只能辅助，不能兜底。

## 4. 根因二：正文被误判成 table，主因是分类规则设计错误

### 现象
在 `translate_with_sanitized_background.py:430-452` 中，判断逻辑是：

- 只要有数字和 `%$()`，就可能判为 table
- 数字多到一定程度，也可能判为 table
- 如果在财务章节里且有数字，也可能判为 table

对应代码：
- `unit_is_table_like(...)` `spikes/11_sanitized_background_translation/translate_with_sanitized_background.py:430`
- `determine_term_usage_mode(...)` `spikes/11_sanitized_background_translation/translate_with_sanitized_background.py:444`

### 根因判断
这是一个**明确的工程分类逻辑问题**。

系统当前把以下三件事混在了一起：

1. 文本里有数字
2. 文本处于财务章节
3. 文本本身是表格/标签/表头

但实际上：
- 财务 narrative 大段正文里也会有很多数字
- 有数字不代表它是 table
- 在 CFO review 里也不代表它就是表格

### 为什么这会伤害质量
一旦判成 `table`：
- style rules 会切到 table 模式
- 术语输出会更偏缩写
- narrative 会被“表格化”处理

结果就是：
- 术语可能更统一了
- 但正文更生硬、更像表格注释

### 结论
> **正文质量变硬的根因之一，不是模型不会写，而是系统先把正文错误地当成了 table。**

### 怎么解决
#### P0 工程方案
重写 classification 逻辑，不要再用“有数字”近似 table。

至少拆成三层：
1. **结构证据优先**：block type、bbox 形态、行列对齐、重复列头
2. **文本证据其次**：是否为短标签、是否存在表头样式
3. **章节信息最后参考**：只做弱信号，不做强判定

#### 额外建议
把 `table` 再拆成：
- `table_label`
- `table_cell`
- `financial_narrative`

而不是一个 `table` 吃掉所有财务内容。

## 5. 根因三：大量 shrink 不是翻译问题，主因是渲染策略太弱

### 现象
`report.json` 里大量 unit 处于：
- `fit_status = font_shrink`

而且比例很低，例如：
- `0.271`
- `0.5`
- `0.55`

见：
- `spikes/11_sanitized_background_translation/output/focus_pages_10_13_19_20_run4/report.json:1`

### 根因判断
这类问题的主因不是 prompt，也不是模型，而是**渲染工程能力不够**。

因为像：
- `概覽 -> Overview`
- `主席報告 -> Chairman's Statement`

本质上并不是“翻得太长离谱”，而是：
- bbox 不适合当前布局模式
- 标题块没有专门渲染策略
- textbox/single_line/group_slots 过于通用
- 系统默认用缩字体去兜底

### 为什么这是工程问题
因为现在系统的逻辑更像：

> 先翻译，再想办法塞进去，塞不下就缩小。

而不是：

> 根据块类型先选最合适的渲染策略，再决定是否需要紧缩表达。

### 结论
> **大面积 shrink 的根因是渲染层策略不足，而不是翻译层本身失控。**

### 怎么解决
#### P0 工程方案
1. 给不同块型单独渲染：
   - heading
   - navigation label
   - signature/name-role
   - narrative paragraph
   - table cell
2. 建立 shrink 阈值：
   - 标题低于某阈值直接判失败
   - 正文连续多段 shrink 判页面高风险
3. 不要只会缩字体：
   - 允许标题短译策略
   - 允许双行结构保留
   - 允许块内重排优先于全局 shrink

## 6. 根因四：提示词承担了过多职责，导致控制不稳定

### 现象
当前 prompt 里同时塞了：
- 场景信息
- 背景信息
- section 信息
- roles
- terms
- style rules
- 当前 unit 内容

对应见：
- `build_context_pack(...)` `spikes/11_sanitized_background_translation/translate_with_sanitized_background.py:541`
- `build_translation_prompts(...)` `spikes/11_sanitized_background_translation/translate_with_sanitized_background.py:582`

### 根因判断
这是一个**prompt 设计问题**，但本质上仍然是“工程没有前移 enough，导致 prompt 被迫背锅”。

当前 prompt 让模型同时完成：
1. 理解企业年报场景
2. 理解当前章节
3. 理解术语规则
4. 理解角色映射
5. 理解风格要求
6. 完成翻译
7. 隐式兼顾布局长度

这对模型来说负担太重。

### 结果
会出现：
- 某些地方术语更稳了
- 某些地方语气却变硬了
- 某些地方数字规则遵守得不稳定
- 某些地方 narrative 被“规则化”得太过

### 结论
> **提示词问题的根因，不是写得太差，而是 prompt 承担了本该由工程层承担的职责。**

### 怎么解决
#### P1 方向
1. prompt 只保留模型真正需要的信息：
   - 当前 unit
   - 极少量必要 context
   - 明确术语映射
2. 把不该放 prompt 的内容前移到工程层：
   - 数字保护
   - 单位锁定
   - 章节类型判定
   - 特殊块渲染决策
3. 让 prompt 负责“表达”，工程负责“约束”。

## 7. 根因五：模型能力存在边界，但不是当前首要矛盾

### 现象
人工反馈指出：
- 人工译文更像正式发布文本
- 系统译文虽然可用，但和人工专业金融译文仍有明显差距

### 根因判断
这里确实有**模型边界问题**，但不是主因。

因为模型已经证明：
- 能产出可用金融英文
- 能遵守部分术语控制
- 能在一定程度上压缩表达

说明模型不是完全做不到。

但它也有明显边界：
- 不能稳定达到资深金融译者的文风掌控
- 不能仅靠 prompt 长期稳定处理所有财务口径细节
- 不能天然兼顾“事实保真 + 文风成熟 + 页面适配”三目标

### 结论
> **模型边界是客观存在的，但当前最该优先解决的不是换模型，而是把该工程化的先工程化。**

### 什么时候才该把问题归到模型
只有当下面这些都做完以后，问题仍明显存在，才应优先怀疑模型：
- 数字已锁定
- 分类已修正
- 渲染已分块优化
- prompt 已减负
- 评测集已稳定

在这之前，贸然归因“模型不行”，会掩盖真正该修的系统问题。

---

## 8. 每类问题该怎么处理

### 8.1 工程化问题（P0，优先最高）
必须先修：
- 财务数字锁定与回校
- table / narrative / heading 分类重构
- shrink 阈值与失败判定
- 标题/署名/栏目名专门渲染

### 8.2 Prompt 问题（P1）
在工程改完一轮后再收：
- 减少无关背景注入
- 精简 style rules
- narrative 与 table 使用不同、更短的 prompt
- 让 prompt 聚焦表达，而不是承担事实保护

### 8.3 模型问题（P2）
最后再评估：
- 是否需要更强模型
- 是否需要专门的金融翻译微调/候选重排
- 是否需要引入人工参考译文做 rerank

---

## 9. 最终判断

如果现在必须非常明确地回答：

### 根因到底在哪？
答案是：

#### 第一根因：工程化问题
- 没有把财务事实锁住
- 没有把正文和表格分开
- 没有把特殊块单独渲染
- 没有把页面可交付标准硬化

#### 第二根因：prompt 承担职责过多
- 该工程做的事放进了 prompt
- 导致模型负担过重，控制不稳

#### 第三根因：模型能力边界
- 存在，但不是当前最先该动的地方

### 最准确的一句话
> **当前系统与人工专业译文之间的主要差距，主因在工程层，次因在 prompt 层，模型能力边界是客观约束但不是当前首要瓶颈。**

---

## 10. 建议执行顺序

### 第一阶段：先修工程
1. 数字/金额/股息口径锁定与回校
2. narrative/table/heading 分类重构
3. 特殊块渲染策略
4. shrink 风险阈值与页面失败判定

### 第二阶段：再修 prompt
1. 缩短 prompt
2. 分块类型定制 prompt
3. 只保留必要背景

### 第三阶段：最后评估模型
1. 修完前两层后重新对比人工译文
2. 如果差距仍大，再判断是否需要更强模型或 rerank 机制
