## 1. 产品定位与核心用户

### 1.1 产品定位

AI翻译系统是一个**专业文档翻译平台**，专注于PDF文档的智能翻译服务。系统采用"工程化优先"的设计理念，通过确定性工程方案与AI能力的精准结合，为用户提供高质量、可控的文档翻译体验。

**核心价值主张：**
- **专业可靠**：80%工程化+20%AI推理，确保翻译质量的稳定性和可预测性
- **过程透明**：实时展示AI执行过程，让用户理解翻译决策逻辑
- **结果可控**：支持术语管理、质量检查、人工审校等多重质量保障机制
- **体验流畅**：渐进式翻译展示，类似KIMI PPT生成的流畅体验

### 1.2 核心用户画像

**主要用户群体：**

1. **企业翻译团队**
   - 需求：批量处理技术文档、产品手册、合同协议等
   - 痛点：传统翻译工具质量不稳定，专业术语处理能力弱
   - 期望：可定制术语库，支持团队协作，翻译结果可追溯

2. **专业译者**
   - 需求：提升翻译效率，保持专业质量
   - 痛点：重复性工作多，缺乏智能辅助工具
   - 期望：AI辅助翻译+人工审校，保留专业控制权

3. **科研工作者**
   - 需求：快速翻译学术论文、技术报告
   - 痛点：专业术语翻译不准确，格式保留困难
   - 期望：保持原文格式，准确翻译专业术语

### 1.3 MVP阶段核心场景

**场景1：单文档翻译**
- 用户上传PDF文档
- 配置翻译参数（源语言、目标语言、术语库）
- 观察实时翻译进度
- 下载翻译结果

**场景2：翻译结果审校**
- 打开已完成的翻译任务
- 逐页对比原文与译文
- 发现问题后终止任务
- 调整术语库后重新翻译

## 2. 整体架构设计

### 2.1 信息架构

```
AI翻译系统
├── 任务管理
│   ├── 任务列表（查看所有翻译任务）
│   └── 创建任务（上传文档、配置参数）
├── 翻译执行
│   ├── 实时进度监控
│   ├── AI执行过程展示
│   └── 渐进式翻译预览
└── 结果审校
    ├── 逐页对比视图
    ├── 质量报告查看
    └── 结果下载
```

### 2.2 页面结构与导航

**主要页面：**

1. **任务列表页** (`/tasks`)
   - 展示所有翻译任务
   - 支持筛选、搜索、排序
   - 快速操作入口（查看、下载、删除）

2. **任务创建页** (`/tasks/new`)
   - 文档上传
   - 参数配置
   - 术语库选择

3. **翻译执行页** (`/tasks/:id/translate`)
   - 左侧：PDF预览区（渐进式翻译展示）
   - 右侧：AI执行过程侧边栏
   - 顶部：进度条与控制按钮

4. **对比审校页** (`/tasks/:id/review`)
   - 左侧：原文PDF
   - 右侧：译文PDF
   - 页面同步滚动
   - 底部：页码导航

### 2.3 布局规范

**通用布局结构：**

```
┌─────────────────────────────────────────┐
│ 顶部导航栏 (Header)                      │
│ Logo | 任务列表 | 创建任务 | 用户信息    │
├─────────────────────────────────────────┤
│                                         │
│                                         │
│          主内容区 (Main Content)         │
│                                         │
│                                         │
├─────────────────────────────────────────┤
│ 底部信息栏 (Footer)                      │
│ 版本信息 | 帮助文档 | 反馈入口            │
└─────────────────────────────────────────┘
```

**响应式设计原则：**
- 最小支持分辨率：1366x768
- 推荐分辨率：1920x1080及以上
- 核心功能（翻译执行、对比审校）不支持移动端（需要大屏幕）

#### 3.1.2 任务状态定义

| 状态 | 显示文本 | 颜色 | 说明 |
|------|----------|------|------|
| pending | 等待中 | 灰色 | 任务已创建，等待执行 |
| parsing | 解析中 | 蓝色 | 正在解析PDF文档 |
| understanding | 理解中 | 蓝色 | 文档理解Agent正在分析 |
| translating | 翻译中 | 蓝色 | 正在执行翻译 |
| checking | 检查中 | 蓝色 | 质量检查中 |
| rendering | 渲染中 | 蓝色 | 正在生成译文PDF |
| completed | 已完成 | 绿色 | 翻译完成 |
| failed | 失败 | 红色 | 翻译失败 |
| paused | 已暂停 | 黄色 | 用户手动暂停 |
| terminated | 已终止 | 橙色 | 用户手动终止 |

#### 3.1.3 交互行为

**筛选功能：**
- 按状态筛选：全部/进行中/已完成/失败
- 按语言对筛选：中→英/英→中/其他
- 按时间筛选：今天/本周/本月/自定义

**排序功能：**
- 创建时间（默认，降序）
- 完成时间
- 文档名称
- 质量评分

**搜索功能：**
- 支持文档名称模糊搜索
- 实时搜索（输入300ms后触发）

**批量操作：**
- 支持多选任务
- 批量下载
- 批量删除

#### 3.3.2 渐进式翻译展示

**设计理念：**

**实现方式：**
1. 初始状态：显示原文PDF
2. 翻译中：当前文本块高亮，译文逐字显示
3. 完成：整页显示译文

**展示内容：**

```
├─ ✓ 提取关键术语 (1.8s)
└─ ✓ 分析翻译难点 (3.1s)

📝 翻译执行 (进行中)
├─ ✓ 第1-11页 (已完成)
├─ ▶ 第12页 (翻译中)
└─ ⏳ 第13-20页 (等待中)

✅ 质量检查 (等待中)
```


#### 3.4.2 页面对齐策略

**核心原则：**
- 1:1页面对应（第N页原文 对应 第N页译文）
- 同步滚动（滚动左侧，右侧同步）
- 页码同步（切换页码，两侧同步）

**实现方式：**
- 使用PDF.js渲染PDF
- 监听滚动事件，计算滚动比例
- 同步应用到另一侧

#### 3.4.3 交互行为

**滚动同步：**
- 滚动任一侧，另一侧同步滚动
- 保持相对位置一致

**页码导航：**
- 点击页码快速跳转
- 支持键盘快捷键（←/→翻页）

**缩放控制：**
- 支持缩放（50%-200%）
- 两侧缩放比例同步

**操作按钮：**
- 下载原文：下载原始PDF
- 下载译文：下载翻译后的PDF
- 重新翻译：返回任务创建页，预填参数

## 6. 技术实现建议

### 6.1 技术栈概览

#### 6.1.1 技术选型对比表

| 技术类别 | 推荐方案 | 备选方案 | 选型理由 |
|---------|---------|---------|---------|
| 前端框架 | React 18 | Vue 3 | 生态成熟，组件库丰富，团队熟悉度高 |
| 状态管理 | Zustand | Redux Toolkit | 轻量级，API简洁，适合中小型项目 |
| UI组件库 | Ant Design 5 | Material-UI | 企业级设计，中文文档完善，组件丰富 |
| 路由管理 | React Router 6 | - | 官方推荐，功能完善 |
| HTTP客户端 | Axios | Fetch API | 拦截器支持，错误处理完善 |
| PDF渲染 | PDF.js | react-pdf | Mozilla官方，功能强大，稳定性高 |
| 实时通信 | EventSource (SSE) | WebSocket | 单向推送场景，实现简单 |
| 代码规范 | ESLint + Prettier | - | 代码质量保障 |
| 构建工具 | Vite | Webpack | 开发体验好，构建速度快 |
| 类型检查 | TypeScript | - | 类型安全，提升代码质量 |

### 6.2 PDF渲染方案

**核心技术：** PDF.js

**关键功能：**
- PDF文档加载与渲染
- 页面导航与缩放
- 文本层叠加（支持文本选择）
- 渐进式内容替换

**实现要点：**
1. 使用Canvas渲染PDF页面
2. 在Canvas上叠加HTML层显示译文
3. 根据文本块坐标精确定位
4. 使用CSS动画实现打字机效果

### 6.3 实时通信方案

**核心技术：** Server-Sent Events (SSE)

**选择理由：**
- 单向推送（服务器→客户端），符合翻译进度推送场景
- 基于HTTP协议，无需额外配置
- 自动重连机制
- 浏览器原生支持

**事件类型设计：**
1. `progress` - 进度更新
2. `translation_chunk` - 翻译内容流
3. `execution_log` - AI执行日志
4. `task_completed` - 任务完成

### 6.4 状态管理方案

**核心技术：** Zustand

**Store设计：**
- `taskStore` - 任务列表、当前任务状态
- `userStore` - 用户信息、偏好设置
- `glossaryStore` - 术语库管理

**优势：**
- API简洁，学习成本低
- 无需Provider包裹
- TypeScript支持良好
- 性能优秀

### 6.5 Mock数据方案

**核心技术：** Mock Service Worker (MSW)

**优势：**
- 在浏览器层面拦截请求
- 无需修改业务代码
- 支持SSE流式响应模拟
- 开发和测试环境通用

**Mock数据覆盖：**
- 任务列表数据
- 任务创建响应
- SSE流式进度推送
- PDF文件URL
- 术语库数据
