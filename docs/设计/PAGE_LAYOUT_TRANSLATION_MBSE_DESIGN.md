# 逐页布局翻译与高保真还原系统设计（MVP）

> 版本：v0.2 MVP  
> 日期：2026-04-13  
> 范围：先打通 `PDF -> 逐页抽块 -> 翻译 -> 回填 -> 输出 PDF` 的关键路径  
> 密钥原则：模型 URL 与 Token 只通过环境变量注入，不在文档中记录真实值

---
 
## 1. 文档目标

本设计只解决一件事：

> 给定一份数字 PDF，逐页提取文本块，翻译后尽量放回原页面对应位置，输出翻译后的目标 PDF。

这份文档是 MVP 设计，不追求一步到位解决全部高保真问题。目标是先证明关键路径成立，再决定后续是否扩展到复杂表格、扫描件、审阅系统和可编辑 DOCX。

---

## 2. MVP 结论

### 2.1 先做什么

MVP 只做：

- 输入数字 PDF。
- 逐页解析文本块和坐标。
- 调用大模型翻译每个文本块。
- 在原页面对应区域覆盖并写入译文。
- 输出新的 PDF。
- 输出逐页 JSONL 报告，记录每块的原文、译文、坐标、是否溢出。

### 2.2 先不做什么

MVP 不做：

- 不做 DOCX 主链路。
- 不做 Markdown 中间格式。
- 不做完整多智能体系统。
- 不做扫描 PDF OCR。
- 不做复杂表格结构重建。
- 不做前端审阅系统。
- 不做全文术语学习闭环。
- 不做跨页自动重排。

### 2.3 为什么这样收敛

当前真正要验证的是：

1. 页面文本块能否稳定抽取。
2. 翻译后文本能否回填到原位置。
3. 长度膨胀是否会导致大面积溢出。
4. “页面背景 + 局部文本覆盖”这条路线是否可接受。

只要这 4 点成立，后续复杂能力再逐步加。

## 3. MVP 系统边界

### 3.1 输入

- 数字 PDF。
- 源语言：中文或英文。
- 目标语言：英文或中文。

### 3.2 输出

- `target.pdf`：翻译后的 PDF。
- `layout_translation_report.jsonl`：逐页逐块报告。

### 3.3 外部依赖

- PDF 解析与渲染库。
- 大模型翻译接口。
- 本地文件系统。

---

## 4. MVP 主链路

```text
Source PDF
  -> Page Iterator
  -> Page Render
  -> Text Block Extractor
  -> Block Translator
  -> Fit Check
  -> Page Overlay Renderer
  -> Target PDF
  -> JSONL Report
```

## 5. 核心处理逻辑

### 5.1 逐页处理

系统必须按页工作，而不是一次性把整本 PDF 抽成纯文本。

每页处理步骤：

1. 渲染当前页面背景。
2. 提取文本块和块坐标。
3. 为每个块生成翻译任务。
4. 调用模型翻译。
5. 检查译文是否能放进原块区域。
6. 回填到当前页。
7. 写入目标 PDF。
8. 记录当前页 JSONL 报告。

### 5.2 为什么必须逐页

- 能保住页边界。
- 能保住大体视觉布局。
- 能单页失败、单页重试。
- 能支持大文档流式处理。

---

## 6. 页数策略

### 6.1 不限制总页数

MVP 不是“只支持 5 页”的原型。

系统设计上必须支持任意页数文档，包括：

- 10 页
- 100 页
- 300 页
- 500 页以上

限制只来自：

- 机器内存
- 模型吞吐
- 任务时间

而不是来自系统结构本身。

### 6.2 如何支持任意页数

采用分页流式处理：

- 一次只加载当前页或当前批次页。
- 每页处理完成立即写出结果。
- 不要求把整本 PDF 全部展开到内存中。

建议批次单位：

- 默认 `1 页`
- 可选 `5~20 页` 批处理，用于提速

### 6.3 断点续跑

MVP 必须预留断点续跑能力：

- 记录已完成页码。
- 失败页可单独重跑。
- 支持 `--pages 1-20`、`--pages 55` 之类的调试参数。

这里的页码参数只是调试入口，不是功能上限。

## 7. 数据模型

### 7.1 PageRecord

```json
{
  "page_no": 1,
  "width": 595.28,
  "height": 841.89,
  "blocks": []
}
```

### 7.2 BlockRecord

```json
{
  "block_id": "p1_b3",
  "bbox": [72.1, 120.4, 410.8, 168.9],
  "source_text": "協助大眾實現健康、長久、好生活。",
  "target_text": "Helping people live Healthier, Longer, Better Lives.",
  "font_size": 11.5,
  "fit_status": "ok",
  "fit_strategy": "none"
}
```

### 7.3 JSONL 报告

每页每块输出一行，最少包含：

- 页码
- 块 ID
- 原文
- 译文
- 坐标
- 字号
- 是否溢出
- 是否缩字号
- 模型调用 ID 或哈希

---

## 8. 大模型职责

### 8.1 MVP 里大模型只做一件主事

大模型只负责：

- 把一个文本块翻译成目标语言，并尽量短、尽量适配原位置

### 8.2 MVP 里大模型暂时不做

- 不做全局排版规划
- 不做表格结构重建
- 不做全书术语学习
- 不做多 agent 自主讨论

### 8.3 模型调用方式

通过 Anthropic-compatible 接口接入：

```text
ANTHROPIC_BASE_URL=<runtime secret>
ANTHROPIC_AUTH_TOKEN=<runtime secret>
```

Prompt 目标：

- 翻译准确
- 保留数字、日期、金额、百分比、公司名
- 尽量简洁
- 不输出解释
- 不输出 Markdown

---

## 9. 是否需要多智能体

### 9.1 MVP 不需要

MVP 阶段不需要复杂多智能体编排。

一个受控流水线就够：

- Extractor
- Translator
- Fit Checker
- Renderer

### 9.2 后续何时再加

后续只有在下面问题明显出现时，才考虑多智能体：

- 块分类错误很多
- 超长文本修复逻辑变复杂
- 术语冲突频繁
- 质检解释需要独立流程

届时再拆：

- TranslationAgent
- FitRepairAgent
- QualityAgent

但这不是 MVP 必需条件。

## 10. 技术选型（MVP）

### 10.1 推荐最小栈

- Python
- PyMuPDF
- requests 或 httpx
- JSONL

### 10.2 选型原因

#### PyMuPDF

用于：

- 打开 PDF
- 提取文本块与坐标
- 渲染页面
- 在页面上覆盖和写入文本
- 输出新 PDF

原因：

- 依赖少
- 一库可先打通 MVP 主链路
- 比先引入一大堆 layout/OCR/Office 组件更轻

#### Anthropic-compatible API

用于：

- 文本块翻译

原因：

- 先把模型调用路径打通
- 后续可切换成内网模型或别的兼容端点

#### JSONL

用于：

- 保存每页每块处理结果

原因：

- 简单
- 易调试
- 易做断点续跑

## 11. 页面回填策略

### 11.1 MVP 方案

对每个文本块：

1. 用浅色或白色背景覆盖原文本区域。
2. 在原 bbox 内写入译文。
3. 默认使用接近原字号。
4. 若放不下，逐步缩小字号。
5. 若仍放不下，标记 `overflow`。

### 11.2 允许的简化

MVP 允许：

- 背景覆盖略显生硬。
- 某些复杂装饰性文字先不翻。
- 少量溢出进入报告。

### 11.3 不允许的行为

- 静默丢失文本。
- 数字、金额、日期被随意改写。
- 没有记录就跳过块。

---

## 12. 质量判断

### 12.1 MVP 成功标准

MVP 路线成立的最低标准：

1. 能处理整本 PDF，而不是只支持固定 5 页。
2. 输出 PDF 页数与输入一致。
3. 大部分正文文本块可成功翻译并回填。
4. 页面背景和大体布局保留。
5. 所有失败块都出现在 JSONL 报告中。

### 12.2 MVP 不要求

MVP 暂不要求：

- 表格页完美翻译
- 封面页完全美观
- 与原 PDF 几乎像素级一致
- 可编辑 DOCX

## 13. 技术穿刺

### Spike 1：文本块抽取

目标：

- 对样本 PDF 输出逐页文本块和 bbox。

验收：

- 前 20 页能稳定抽取。
- 整本也能跑完。
- 每页块数、块坐标、原文都可保存。

### Spike 2：块翻译

目标：

- 每个块能调用模型翻译。

验收：

- 保留数字、金额、年份。
- 译文不带解释。
- 输出与 block 对齐。

### Spike 3：PDF 回填

目标：

- 把译文写回原页面。

验收：

- 输出 PDF 页数一致。
- 背景保留。
- 正文页能读。

### Spike 4：整本流式处理

目标：

- 支持任意页数 PDF 顺序处理。

验收：

- 不把整本所有页面一次性加载到内存。
- 支持分页断点。
- 支持失败页重跑。

## 15. 命令形态建议

```powershell
$env:ANTHROPIC_BASE_URL="..."
$env:ANTHROPIC_AUTH_TOKEN="..."

python spikes\page_translate_mvp\translate_pdf.py `
  --input 样本\中文\AIA_2021_Annual_Report_zh.pdf `
  --output spikes\page_translate_mvp\output\aia_2021_en.pdf `
  --report spikes\page_translate_mvp\output\layout_translation_report.jsonl `
  --target-lang en
```

可选参数：

- `--pages 1-20`：仅调试指定页范围
- `--resume`：从断点继续
- `--batch-size 5`：分页批处理

这里 `--pages` 只是调试，不代表系统只能翻固定页数。

## 17. 最终建议

当前最合理的路线不是一开始做复杂全栈系统，而是先做一个简单但完整的 MVP：

```text
PDF
  -> 逐页抽块
  -> 块级翻译
  -> 回填原页面
  -> 输出翻译后 PDF
  -> 输出逐页报告
```

这条链路一旦打通，就已经能回答最关键的问题：

- 高保真翻译回填是否成立
- 页级布局是否可保
- 任意页数文档是否能稳定处理
- 大模型是否适合做块级翻译

MVP 阶段不追求复杂，只追求关键路径打通。
